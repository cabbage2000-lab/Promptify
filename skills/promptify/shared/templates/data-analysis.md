# Data Analysis Template

## Brief

目标：
把模糊的"分析一下数据"重述为一个可验证的分析问题：明确要回答什么、用哪些数据、结论以什么口径和阈值判定成立——产出带关键数字的明确结论，而非泛泛的描述性概览。

假设：
数据分析充满隐含前提：时间范围、数据粒度（一行代表什么）、口径定义（"活跃""转化"怎么算）、去重与过滤规则、是否含测试或内部流量。动手前显式写明这些假设；探索后仍无法确认的，在结论里标注，不要静默选一种口径就出数。

模式：
先生成高质量 brief 并询问是否进入执行阶段；含高风险信号时先分析并确认。

上下文：
按 `shared/context-discovery.md` 做最小必要发现，先摸清数据再算数：数据源与访问方式（文件 / 数据库表 / 数仓 / 日志 / 埋点）、表结构与字段含义、样本行与行数量级、关键字段的空值率与取值分布、时间覆盖范围、已有口径定义与现成查询或 notebook。沿用项目既有分析工具栈（SQL / pandas / DuckDB / notebook 等），不另起炉灶。

要求：
先理解数据的粒度与口径，再写查询或脚本回答所问问题，并保持可复现（查询/脚本存档，而非只在脑中或一次性粘贴）。非平凡分析须做数据完整性自检：行数量级是否合理、关键字段空值率、是否重复计数、join 后基数是否爆炸或丢行、时间是否连续覆盖、极端值是否污染统计；明显一次性的小探索可相应放宽，但要说明放宽了什么。结论须不过度解读：给出样本量，区分相关与因果，标注口径与局限，不把噪声当趋势。范围只限所问问题，不顺手扩成大盘报告，不引入无关数据源或新工具。最终汇报：结论与关键数字、所用数据与口径、复现方式（查询/脚本/参数）、做过的完整性校验、以及结论的局限与待澄清项。
