# Evolve Template

## Brief

目标：
把"把 X 调好/优化下 X"重述为一个指标驱动的进化优化三元组：被进化对象（哪个文件或产物）、评估契约（指标定义与测量方式）、目标值或预算（默认代数上限 3）。交付物是达标或预算用尽后的最优代产物、含基线的完整分数曲线与复跑方式，而非一次性改写。

假设：
进化优化依赖显式评估契约。用户未给出评分方式时，在此提出候选方案并标注待确认：优先设计可执行评分命令（读取候选产物，输出含 score 字段的 JSON，分数越高越好）；确无客观指标时退化为固定 rubric 的 LLM 评分（每代同一评审、rubric 写入 brief、结论标注主观性局限）。默认停止条件：达到目标值、代数达上限、或连续 2 代无提升；与用户预算不符时在此修改。

模式：
先生成高质量 brief 并询问是否进入执行阶段。进化循环是多代多 agent 的高成本操作：确认执行前须告知将在目标项目创建 .claude/agents/evolve-meta.md、.claude/agents/evolve-feedback.md 与 .claude/workflows/ 循环脚本，经用户同意后才启动；评估命令含高风险信号时先分析并确认。

上下文：
按 `shared/context-discovery.md` 做最小必要发现：被进化产物的当前内容与位置、现有评估手段或测试命令、运行环境与依赖；执行开始时先对原始产物跑一次评估，记为 gen 0 基线分数。编排契约、角色定义骨架与代际产物布局按 `shared/evolution-loop.md`。

要求：
按 `shared/evolution-loop.md` 搭建代际循环：evolve-meta 生成首代；每代执行评估契约、把分数、变更摘要与经验教训追加进 history.md；evolve-feedback 基于上一代产物、score.json 与全部历史写 improvement.md 和下一代，不重复历史中已失败的尝试。进化只发生在 `evolve-runs/run_<id>/gen_<n>/` 下的副本上，不直接修改原始文件；范围只限被进化对象，不顺手扩展到无关文件。最终汇报：含 gen 0 基线的分数曲线、最优代及其相对基线的差异摘要、各代 improvement.md 要点、复跑与续跑方式（workflow 脚本路径与参数）；经用户明确确认后才把最优代回写原始文件。
